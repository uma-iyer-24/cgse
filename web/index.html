Metric glossary (how to interpret the logs)

Field Meaning Why it matters
val_acc Accuracy on the CIFAR-10 test split (we call it “val”). Primary performance metric; comparable across arms within a tier.
num_parameters Trainable parameter count after any structural edits. Budget / capacity; report alongside accuracy.
teacher_forwards Cumulative teacher forward passes during training. Proxy for extra compute cost of KD/teacher-guided methods. CGSE teacher-free rows have 0.
wall_seconds Cumulative wall-clock seconds since run start. Supports accuracy-per-time comparisons and practical efficiency claims.
train_steps Cumulative number of training batches processed. Normalizes compute across hardware and helps compare “work done”.
epoch_seconds Wall time for an epoch (train + eval). Helps identify overhead and compare per-epoch cost.

Where CGSE beats Existing Systems

  • Teacher-free evolution: CGSE can run with teacher_forwards = 0 while still evolving structure.
  • Lower auxiliary compute: KD/teacher baselines pay teacher inference every training step; we measure it directly as teacher_forwards and as added wall_seconds.
  • Accuracy-per-time: compare val_acc as a function of wall_seconds (or GPU-hours) rather than only final accuracy.
  • Evolution efficiency: for multi-stage runs, report edits-to-threshold using mutation JSONL.
